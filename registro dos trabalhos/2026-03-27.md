## 🛠️ O Que Foi Feito Hoje (Detalhamento Técnico)

### 1. Desbloqueio da Infraestrutura Appwrite
- **Problema**: O script de setup inicial falhava com erro `401 Unauthorized`. As credenciais de login manual (`a2tickets360@gmail.com`) não estavam sendo aceitas pelo console remoto.
- **Intervenção**: Migramos o fluxo de autenticação do script (`full_setup_new_server.cjs`) para usar a **Master API Key**.
- **Por que**: Chaves de API de servidor ignoram desafios de login de interface e garantem que o setup seja automatizado e resiliente a mudanças de senha no console.

### 2. Superação do Limite de Linha (MariaDB Row Size Limit)
- **Problema**: A criação da coleção de `events` travava ao chegar no 6º atributo. O servidor retornava `400: Attribute limit exceeded`.
- **Diagnóstico**: O Appwrite utiliza MariaDB/MySQL. O limite de uma linha na tabela é de 65.535 bytes. Com campos `VARCHAR` (Strings) de 15.000 caracteres em `utf8mb4` (4 bytes por char), a linha excedia o limite físico do banco rapidamente.
- **Solução**: Redimensionamos o campo `description` de 15.000 para **5.000** caracteres e URLs de 2.000 para **500**.
- **Resultado**: Conseguimos criar todos os **22 atributos** da coleção de eventos, garantindo que o banco comporte todos os dados (datas, localização, fotos, status de pagamento) em uma única estrutura eficiente.

### 3. Provisionamento de Schema e Verificação
- **Escopo**: 14 coleções e 3 Buckets de Storage criados e validados.
- **Auditoria**: O script `_verify_appwrite.cjs` foi executado ao final, confirmando que o schema está **"Ready for Production"** no servidor `database.a2tickets360.com.br`.

## 🚀 Roteiro de Continuidade (O Que e Por Que)

Para amanhã, seguiremos este cronograma detalhado para transformar o backend monolítico em uma arquitetura modular e moderna:

### **Fase 1: Singleton do SDK do Appwrite**
*Arquivo: `server/src/lib/appwrite.ts`*
- **O que**: Configurar o cliente oficial do `node-appwrite` exportando instâncias de `Databases`, `Storage` e `Users`.
- **Por que**: Evitar a criação de múltiplas conexões com o servidor (overhead) e centralizar a autenticação da API Key em um único lugar seguro.

### **Fase 2: Modularização da Autenticação (A Cirurgia)**
*Arquivos: `routes/auth.ts` e `routes/organizers.ts`*
- **O que**: Extrair os blocos de autenticação do arquivo `index.ts` (atualmente com 1620 linhas).
- **Por que**: O arquivo `index.ts` está "obeso", o que torna a manutenção perigosa e a depuração lenta. Separar em rotas modulares permite que cada parte do sistema (Login, Cadastro de Produtor, Perfil) seja testada e escalada individualmente.

### **Fase 3: Migração de Persistência (100% Appwrite)**
- **O que**: Substituir as chamadas do Drizzle/Postgres (`db.insert`, `db.select`) pelas chamadas nativas do Appwrite.
- **Por que**: Padronizar o ecossistema. O Appwrite oferece recursos integrados (Real-time, Auth, Storage) que são mais complexos de gerenciar manualmente no Postgres. Manteremos o Postgres apenas como backup temporário durante a transição fluida.

### **Fase 4: Sincronização de Pagamentos (Asaas)**
*Serviço: `server/src/services/asaas.ts`*
- **O que**: Atualizar os Webhooks de pagamento para salvar o status de "Paid" ou "Pending" diretamente na coleção `sales` do Appwrite.
- **Por que**: Garantir que o produtor veja as vendas em tempo real no dashboard, utilizando a alta performance de escrita do Appwrite.
