# Registro de Trabalhos - 23 de Junho de 2026

## 🎯 Objetivo Principal
Corrigir o compartilhamento de links de eventos no WhatsApp e Redes Sociais, que estavam exibindo a imagem padrão da plataforma "Lovable" ao invés do banner e das informações reais do evento da A2 Tickets 360º.

## 🛠️ O que foi implementado e corrigido

### 1. Limpeza do Frontend (`index.html`)
- Remoção completa de todas as meta tags do Lovable (`og:image`, `twitter:image`, etc).
- Substituição pelas meta tags e imagens padrão da **A2 Tickets 360º** (`logo_512x512.png`), servindo como fallback global.

### 2. Criação de Rota Dinâmica no Backend (Hono API)
- Criada a nova rota `GET /og/events/:id` em `server/src/index.ts`.
- Essa rota intercepta a requisição, busca os detalhes do evento no banco de dados e retorna um HTML estático contendo as meta tags dinâmicas do evento (título, descrição com data e local, e o `banner_url`).
- Incluído um sistema de "fallback" de segurança que, em caso de erro na busca do evento, devolve os dados e a logo padrão da A2 Tickets.

### 3. Interceptação de Crawlers no Nginx (`.docker/nginx.conf.template`)
- Configurado um bloco `map` para identificar os `User-Agents` dos principais robôs de redes sociais (WhatsApp, Facebook, Twitter, LinkedIn, etc).
- Implementada a interceptação inteligente na rota `/events/*`:
  - Se for um usuário normal: A requisição segue para a SPA (React).
  - Se for um robô (crawler): A requisição é redirecionada para a rota `/og/events/:id` da API.
- **Correção de Bug do Nginx:** Corrigido o problema clássico de usar `proxy_pass` dentro de um bloco `if` (que estava causando `404 Not Found`). Foi implementado o padrão `error_page 418 = @crawler` com `rewrite` seguro.

### 4. Correção Crítica de Schema e Banco de Dados (`server/src/index.ts`)
- **Fix de Schema do Drizzle:** Descobrimos que a definição do schema no TypeScript (`server/src/db/schema.ts`) estava dessincronizada com as colunas reais do banco de dados Postgres no Supabase (ex: `date` vs `start_date`, `image_url` vs `banner_url`). Isso fazia com que o Drizzle ORM desse crash na consulta e caísse no fallback (mostrando a logo da A2).
- **Bypass com SQL Puro:** Para resolver o problema da rota OG sem precisar de grandes migrações de banco que poderiam quebrar outras partes do sistema, a busca do evento na rota `/og/events/:id` foi reescrita utilizando SQL bruto (`db.execute(sql'...')`), puxando exatamente as colunas corretas.
- **Fix de ReferenceError Global:** Identificadas e corrigidas chamadas que geravam `ReferenceError` na API. Foram injetadas as desestruturações corretas do schema (`const { events, admins, ... } = schema;`) e adicionadas as importações faltantes de operadores do Drizzle (`or`, `and`, `sql`, `isNull`, `inArray`).

## ✅ Resultados
- Testado e validado no **Facebook Sharing Debugger**.
- O sistema agora lê corretamente os links dinâmicos e exibe a arte visual (`banner_url`) cadastrada pelo produtor, juntamente com o título do evento e a descrição otimizada.
- O cache do WhatsApp manterá a imagem antiga temporariamente para links antigos, mas links novos já funcionarão instantaneamente.
